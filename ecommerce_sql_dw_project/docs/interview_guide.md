# 面试讲解指南

## 1. 30 秒项目介绍

我做了一个基于 MySQL 的电商订单数据仓库小项目。项目模拟电商订单数据，从 CSV 原始订单导入开始，设计了 STG 原始层、DIM 维度层、DWD 明细层、DWS 汇总层和 ADS 应用层。项目中使用 SQL 完成 GMV、订单量、支付用户数、客单价、优惠券使用率、SKU 销售排行、平台销售趋势、日环比和 7 日移动平均等指标分析，并通过索引和 EXPLAIN 做了简单查询优化。

## 2. 为什么要分层？

数据分层可以降低复杂度并提高可维护性。STG 层保存原始数据，便于追溯；DWD 层清洗并统一明细口径；DWS 层提前聚合常用指标；ADS 层直接服务报表或业务应用。

## 3. 为什么要有维度表？

订单表中通常只有 `sku_id`、`user_id`、`store_id` 等编号。若要按商品类别、品牌、用户年龄段、门店区域分析，就需要关联维度表补充分析属性。

## 4. LEFT JOIN 后数据变多是什么原因？

常见原因是右表关联键不唯一，导致一对多匹配。例如一个 `sku_id` 在 `dim_sku` 中出现多条记录，订单表关联商品表后会出现行数膨胀。解决方法是检查维度表主键唯一性或先去重。

## 5. LAG 是不是日环比？

不是。`LAG(total_gmv, 1)` 只是取上一条记录的 GMV。真正的日环比增长率是：

```text
(当前 GMV - 上一日 GMV) / 上一日 GMV
```

## 6. 7 日移动平均需要注意什么？

`ROWS BETWEEN 6 PRECEDING AND CURRENT ROW` 表示当前行和前 6 行，一共最多 7 行。只有每天都有数据时，它才近似等价于 7 日移动平均。如果日期缺失，需要先补齐日期。

## 7. 索引怎么解释？

对于经常按平台和日期查询的场景，可以建立联合索引：

```sql
CREATE INDEX idx_dws_platform_date
ON dws_platform_daily(platform, stat_date);
```

它可以支持：

```sql
WHERE platform = 'JD'
  AND stat_date BETWEEN '2025-06-01' AND '2025-06-30'
```

索引不是越多越好。索引会提升查询效率，但也会增加存储成本和写入维护成本。
